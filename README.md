# RadioGate (LILYGO T3 v1.6.1)

LoRa/Wi-Fi шлюз для обмена сообщениями между локальными Wi-Fi клиентами и LoRa-нодами.

## Цель

Сделать систему для **LILYGO T3 v1.6.1** с рабочим диапазоном **433–470 МГц**, с упором на:
- оптимизацию по памяти/нагрузке;
- масштабирование между нодами;
- архитектуру на основе ООП.

---

## Аппаратная платформа

- Плата: **LILYGO T3 v1.6.1 (ESP32 + SX1276/78)**
- Тип связи:
  - локально: Wi-Fi AP + UDP
  - между нодами: LoRa
- Рекомендуемый рабочий диапазон: **433–470 МГц**

> Частота выбирается по региональным требованиям и настройке модуля.

---

## Текущая реализация (в репозитории)

Файл прошивки: `sketch_apr14a.ino`

Реализовано:
- Wi-Fi AP (`LoRa-Hub-1`)
- UDP прием/рассылка локальным клиентам
- AES-256 (CBC) шифрование сообщений перед LoRa отправкой
- прием LoRa, дешифрование и ретрансляция в локальную сеть
- вывод сообщений на OLED SSD1306
- простая таблица пользователей по IP

Текущая частота в коде:
- `LoRa.begin(433E6)`

Для диапазона 433–470 МГц можно менять значение, например:
- `433E6`, `434E6`, `446E6`, `470E6`

## Оптимизированная ООП-архитектура (рекомендуемая)

Чтобы упростить развитие проекта, систему стоит разделить на модули/классы.

### 1) `RadioNodeApp`
Главный orchestration-класс:
- инициализация всех подсистем;
- цикл обработки событий;
- контроль состояния ноды.

### 2) `LoRaService`
Ответственность:
- инициализация LoRa и установка частоты (433–470 МГц);
- отправка/прием пакетов;
- параметры радиоканала (мощность, BW, SF, CR);
- контроль ошибок и RSSI/SNR метрик.

### 3) `CryptoService`
Ответственность:
- шифрование/дешифрование сообщений;
- управление ключами и IV;
- валидация размеров буфера и padding.

### 4) `UdpGateway`
Ответственность:
- прием пакетов от Wi-Fi клиентов;
- отправка сообщений всем активным клиентам;
- буферизация и защита от переполнения.

### 5) `ClientRegistry`
Ответственность:
- хранение списка клиентов;
- выдача/поиск username;
- TTL/lastSeen и очистка неактивных записей.

### 6) `DisplayService`
Ответственность:
- буфер строк OLED;
- безопасный вывод длинных сообщений;
- индикация состояния ноды и ошибок.

## Оптимизации, которые стоит соблюдать

- Минимизировать `String` в горячем цикле (предпочтительно фиксированные буферы).
- Не делать блокирующие `delay()` в критичных участках.
- Ограничить максимальный размер полезной нагрузки LoRa.
- Ввести очередь исходящих сообщений и rate-limit.
- Добавить антидубли пакетов (packetId + TTL).
- Разделить transport-логику и бизнес-логику по классам.

## Базовая настройка частоты

1. Откройте `sketch_apr14a.ino`.
2. Найдите `LoRa.begin(433E6)`.
3. Установите частоту под ваш канал в диапазоне **433–470 МГц**.
4. Прошейте ноду и проверьте обмен между двумя устройствами.
